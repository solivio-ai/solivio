## Phase 1 â€” Data Foundation - DONE

Goal: a working schema, module scaffold, and the canonical JSON format that every future adapter targets.

- Module scaffold (`package.json`, `tsconfig.json`, `src/index.ts`, `src/services.ts`)
- DB schema (`src/data/schema.ts`):
  - `knowledge_base_spaces` â€” top-level navigation unit; each space has its own dedicated map canvas (name, description, color, icon, source metadata: origin, external_id, synced_at); maps to a Confluence Space / Notion workspace / Google Drive folder when adapter-synced

  - `knowledge_base_connections` â€” typed links between articles (from_id, to_id, type: `related|prerequisite|contradicts|...`)
  - `knowledge_base_tags` â€” flat tags on articles (m2m join)
  - `knowledge_base_embeddings` â€” embedding vector per article (article_id, model, halfvec column via the custom `halfvec` Drizzle type from `modules/catalog/src/data/halfvec.ts`) â€” stubbed now, populated in Phase 4
- Migration (`src/data/migrations/0000_init.sql`) via `yarn db:generate knowledge-base`
- Canonical JSON import format definition (TypeScript types + Zod schema in `src/lib/importSchema.ts`): mirrors the DB shape so Confluence/Notion adapters just need to map to this format
- Basic `KnowledgeBaseService` stub in `src/services.ts`: `listCategories`, `listArticles`, `findArticleById`, `upsertFromImport`, `createSpace`
- Events skeleton (`src/events.ts`): `knowledge-base.space.created`, `knowledge-base.article.created`, `knowledge-base.article.updated`, `knowledge-base.import.completed`
- ACL (`src/acl.ts`): `knowledge-base.view`, `knowledge-base.edit`, `knowledge-base.import`

Deliverable: `yarn db:check` passes, module registers, no UI yet.

## Phase 2 â€” Map View (PoC) - DONE

Goal: something on the screen to react to. Raw interactive canvas with the real data â€” good enough to feel the UX, not production-ready.

**Libraries (decided):**
- `@xyflow/react` v12 â€” canvas, pan/zoom, node rendering, edge drawing, free drag placement
- `elkjs` â€” auto-layout engine (layered algorithm); called only when user triggers "Auto-layout"; async so it won't block the UI. Dagre was considered but elkjs handles variable tile sizes and typed cross-link edge routing better â€” matters once users freely add connections between articles.

**Navigation model** (from concept design â€” interpreted loosely; sources section deferred to Phase 7 adapters):
- Left sidebar: spaces list only (no sources section yet â€” external adapters are out of scope for this ticket); clicking a space switches to its dedicated map
- Each space is its own map canvas â€” not a section within a shared canvas
- `+ New space` available in the sidebar at all times, opens a name/color/icon picker, creates the space and navigates into its (empty) map immediately
- Top bar: Map | List toggle, Auto-layout button, `+ Add document`

- Single page `/knowledge-base` â€” sidebar + main React Flow canvas area
- Space map canvas (PoC tier):
  - Pan + zoom + free node drag (React Flow built-in)
  - Article tiles: custom RF node component â€” title, type badge (Article / Directive / Template), body preview, updated timestamp
  - Parent-child hierarchy visible via edge style distinction (vs. typed cross-links)
  - Connection edges: React Flow edges, straight for PoC
  - Click tile â†’ article detail drawer (title + body, read-only)
  - "Auto-layout" button â†’ runs elkjs layered algorithm, updates all node positions, saves to DB
  - Drag node â†’ position saved to DB on drop
- `+ New space` in sidebar â†’ creates empty space, navigates into it immediately
- Seeded with a handful of hardcoded articles across 2 spaces so there's something to compare
- Nav entry + sidebar icon (`src/nav.tsx`)

Exploration questions to answer after this phase:
- Should sub-groups (parent articles with children) get a visually distinct node style, or just a regular tile?
- What connection types warrant visible edge style differences in the PoC vs. Phase 6?
- What information density lands on the tile vs. behind a click?

## Phase 4 â€” JSON Importer + Background Jobs

Goal: real data in, structured the way future adapters will use.

- Importer capability (`src/ai/importers.ts`) â€” accepts the canonical JSON format from Phase 1; maps to `upsertFromImport`; designed so Confluence/Notion/Drive adapters just need to produce that JSON
- Import job (`src/jobs/import.ts`): validates â†’ upserts spaces/articles/connections/tags â†’ generates embeddings (pgvector via `getAi().embeddingModelId()`) â†’ emits `knowledge-base.import.completed`
- Upload UI: file picker in the list view header, progress/status feedback
- Embedding generation wired into every article save (create/edit) as well
- `yarn db:generate knowledge-base` for embedding column migration

## Phase 5 â€” Agent Tools (RAG)

Goal: agents can discover and retrieve knowledge.

- `src/ai/tools.ts` â€” contributes to the global agent-tool registry:
  - `search_knowledge_base(query, opts)` â€” hybrid: embedding similarity + keyword fallback, returns ranked article excerpts
  - `browse_knowledge_base(spaceId?, parentId?)` â€” structural browse, returns article tree with summaries; lets an agent find where to look before pulling content
  - `get_article(articleId)` â€” full article body + connections
- Agent context hint wired into the offer-chat agent (via the merged registry) so the chat agent knows the KB exists and when to reach for it
- `docs/agents.md` updated with the new `knowledge_base_search` / `knowledge_base_browse` tools
